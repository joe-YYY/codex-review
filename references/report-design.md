# HTML 报告设计规范

这份文件约束 Codex Review 的 HTML 报告长什么样。`SKILL.md` 负责流程，本文件负责页面结构、样式和文案。

## 页面目标

报告不是聊天记录，也不是审计日志。它应该帮助用户在 3 分钟内看懂：

- 这段时间 Codex 主要用在了哪里。
- 哪些任务消耗最大。
- 哪些地方造成返工。
- 下周最应该改哪几个提问习惯。
- 是否值得把某些工作方式沉淀成模板、skill、脚本或自动化。

## 固定板块

1. **Header**
   - 标题：`Codex 使用体检报告`
   - 时间范围
   - 生成时间
   - 右上角说明按钮

2. **Hero Insight**
   - 一句话结论。
   - 必须指出最重要的问题或机会，不要写泛泛总结。

3. **Overview**
   - 4-6 张数字卡片。
   - 推荐指标：活跃时长、主项目数、产物数、Token 大致消耗、Prompt 清晰度、返工指数。

4. **Task Mix**
   - 展示任务类型分布。
   - 可用条形图、标签、小卡片，不要求复杂图表。

5. **Project Reviews**
   - 按项目折叠。
   - 默认展开最重要的 1-3 个项目。
   - 每个项目包含：做了什么、耗时、关键产物、低效点、Prompt 诊断、下次推荐 Prompt。

6. **Prompt Diagnosis**
   - 展示跨项目共性 Prompt 问题。
   - 不重复项目折叠区里的“下次推荐 Prompt”。

7. **Habit Review**
   - 总结本周使用习惯。
   - 不给用户贴标签，只描述行为和调整动作。

8. **Personal Profile**
   - 展示主要用途、有效做法、下周调整和当前摩擦。
   - 不要写成性格标签。

9. **Copyable Prompt Templates**
   - 给通用模板，例如复杂任务起手、UI 探索、表格/策略产物。
   - 模板要能直接用于下一次任务。

10. **Productization**
   - 说明本周哪些流程值得沉淀成 skill、脚本、模板或 GitHub 项目。
   - 用 2-3 个阶段卡片展示，不写长段说明。

不要把 `Automation`、`Appendix`、`三色诊断` 放进主报告。自动化是否存在属于任务状态，可以在对话摘要或用户要求时说明；颜色含义只放在右上角说明按钮里；估算说明只保留必要小字，不单独占一节。

## 视觉规则

- 背景用浅灰，内容卡片用白色。
- 主色使用蓝色或绿色；黄色表示需要调整；红色表示容易返工或应避免。
- 不要大面积深色背景。
- 不要使用无意义背景图、装饰大图、过度渐变。
- 卡片圆角 8-14px，阴影要轻。
- 页面宽度建议 1040-1120px。
- 第一屏必须出现标题、时间、颜色说明按钮、一句话结论和关键数字。
- 移动端必须可读：卡片改为单列，表格必要时改为列表或允许横向滚动。
- Prompt 模板必须有复制按钮，或者至少使用深色代码块样式。

## 文案规则

- 结论先行。
- 标题短，尽量 2-6 个字。
- 不写长篇解释。
- 不展示内部调试术语。
- 不写页面自我解释类句子，直接用清晰标题和状态表达。
- 不写低价值备注。用户只需要知道大概消耗、主要问题和下一步动作。
- 少用专业词；必须使用时，用一句人话解释。
- Token 只作为大致消耗展示，不写成账单或效果评分。

## 数据不足时

数据不足也要生成可读报告，不要空白。

推荐空状态：

- `主要产出在对话结论中`
- `本周没有明显高风险提示词问题`
- `状态正常，无需处理`
- `未读到可用 Token 记录`

不要写调试式空状态，例如“字段不存在”“列表为空”“解析失败”。
