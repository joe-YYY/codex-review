# HTML 报告设计规范

报告要让第一次使用的人在 3 分钟内看懂：时间花在哪里、哪些地方发生返工、下周先改什么、哪些流程值得复用。

## 阅读顺序

1. **Header**：标题、时间范围、生成时间、右上角颜色说明。
2. **本周结论**：一句话指出最大投入和最优先改进项。
3. **总览**：活跃时长、项目数、产物数、Token 大致消耗。
4. **效率信号**：提问完整度、方向修正、产物覆盖和主要会话。
5. **任务分布**：按通用任务类型展示耗时、会话和产物。
6. **项目投入**：展示主要项目、占比、Token 和简短观察。
7. **项目复盘**：做了什么、关键产物、效率观察、下次先补什么。
8. **Prompt 诊断**：汇总跨项目最常缺失的信息，不重复完整 Prompt。
9. **本周协作方式**：保持项、优先调整项、下周动作。
10. **使用画像**：只描述本周用途、投入方式和沉淀方式，不做性格标签。
11. **下周可复制 Prompt**：按任务类型去重，最多展示 3 个。
12. **可复用机会**：根据会话数和产物数动态推荐模板、脚本、skill 或自动化。

异常诊断只在有内容时放到页面底部的折叠区域。不要把自动化状态、附录或单独的颜色诊断做成正文板块。

## 视觉规则

- 浅灰背景、白色内容区、蓝色主操作色。
- 绿色表示保持，黄色表示可优化，红色表示优先调整。
- 颜色含义只放在右上角说明按钮。
- 卡片圆角不超过 8px，阴影轻，页面宽度 1040-1120px。
- 第一屏必须出现结论、总览和至少一组效率信号。
- 长路径允许换行；表格在窄屏允许横向滚动。
- 桌面使用多列卡片，移动端使用两列或单列。
- Prompt 使用深色代码块，但页面不能大面积使用深色背景。
- 不使用装饰渐变、背景大图、悬浮色块或无意义动画。

## 文案规则

- 结论先行，标题简短。
- 重点数字和动作使用加粗或状态色，不写连续大段黑色正文。
- 不写解释页面定位或为板块辩解的防御性文案。
- 不写无法帮助用户理解数据或采取行动的备注。
- 专业词只有影响理解时才增加一次轻量提示，不批量堆 tooltip。
- Token 只展示大致消耗，不做质量或效果评分。
- 项目里的“下次先补”只列缺失信息；完整 Prompt 统一放到模板区并去重。

## 数据不足与异常

数据不足时仍生成报告，使用普通用户能理解的状态：

- `主要成果保留在对话结论中`
- `本周没有足够的项目数据`
- `未读到可用 Token 记录`

异常信息必须说明发生了什么、报告是否还能使用、下一步怎么处理。正常状态不显示技术清单。
